### Problem / motivation

`RedisClient` and `PgPool` in `@streetjs/core` only support a single connection endpoint:

- `RedisClientOptions` (`packages/core/src/transports/resp.ts`): `{ host?, port?, password? }` — no `nodes[]`/`sentinels[]` field, and no handling of Redis Cluster `MOVED`/`ASK` redirects.
- `PgConnectOptions` (`packages/core/src/database/wire.ts`): `{ host, port, user, password, database }` — single endpoint only, no replica/standby list, no failover logic.

This means any StreetJS deployment that puts Redis behind **Redis Cluster** or PostgreSQL behind an **HA/failover setup** (Patroni, PgBouncer + standby, RDS Multi-AZ endpoint swaps, etc.) is running against an unsupported topology at the client level — the client has no way to discover cluster slots, follow a `MOVED` redirect, or reconnect to a new primary after a failover.

This gap was found during a release-readiness audit while attempting to close a "Redis Cluster / PostgreSQL HA — NOT VERIFIED" item. It could not be closed by writing a test (as was done for the NATS transport, see `packages/plugin-nats/test/broker.it.test.mjs`) because there's no client capability to test — the gap is in the client, not in test coverage. Tracked in `plans/OUTSTANDING-ACTIONS.md` (item 30).

### Proposed solution

- `RedisClientOptions`: accept an optional `nodes: Array<{host, port}>` (cluster) and/or `sentinels: Array<{host, port}> + name` (sentinel) alongside the existing single `host`/`port`. Implement `CLUSTER SLOTS`/`MOVED`/`ASK` redirect handling when `nodes` is supplied.
- `PgConnectOptions`: accept an optional `hosts: Array<{host, port}>` list with failover/retry-on-connect semantics (try each in order, or use a supplied topology-discovery hook), so a primary failover doesn't require redeploying the app with a new single `host`.
- Both should remain fully backward compatible: the existing single `host`/`port` shape keeps working unchanged when the new fields are omitted.
- Add live integration tests analogous to the existing RabbitMQ/Kafka/NATS ones (`packages/core/src/integration/{rabbitmq,kafka}/*.integration.test.ts`, `packages/plugin-nats/test/broker.it.test.mjs`): gate on broker/cluster availability, skip cleanly (never fail) when unreachable, and exercise a real Redis Cluster (e.g. `docker run grokzen/redis-cluster` or `redis-cli --cluster create`) and a real Postgres failover (e.g. Patroni or a manual primary/standby promote) once the client supports it.

### Alternatives considered

- Document the single-endpoint limitation and leave it unsupported — viable short-term (and is effectively the current state), but blocks any user running a HA/clustered data tier, which is common in production.
- Recommend a third-party client (ioredis/node-redis, pg) for cluster/HA use cases instead of extending the built-in ones — contradicts the framework's zero/near-zero-dependency design goal (native wire drivers), so extending the native clients seems more consistent with the project's direction.

### Backward compatibility

Backward compatible — new fields are additive and optional; existing single-endpoint configs are unaffected.
